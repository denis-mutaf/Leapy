# Архитектурные решения (DECISIONS)

> Здесь объясняем: почему выбрали X, а не Y. Самый ценный файл со временем — без него команда будет наступать на те же грабли.

## [2026-03] Anthropic Claude для анализа, OpenAI для всего остального

**Решение:** Транскрипция и embeddings — OpenAI. Анализ звонков — Anthropic Claude.

**Почему:**
- Claude значительно лучше работает с длинными текстами на русском/румынском и выдаёт более качественные резюме в стиле «живого» CRM-комментария
- OpenAI Whisper (gpt-4o-mini-transcribe) дешевле и быстрее для задачи транскрипции
- text-embedding-3-small — отличное качество/цена для RAG

**Альтернатива, которую рассматривали:** Всё через OpenAI GPT-4o. Отказались — качество резюме хуже.

## [2026-03] Два параллельных вызова Claude вместо одного

**Решение:** `summarizeCall` и `evaluateManager` запускаются через `Promise.all`.

**Почему:**
- Разные задачи требуют разных промптов и контекстов. Объединение в один вызов ухудшает качество обоих.
- Параллельность экономит ~1-2 сек на каждом звонке
- Независимые ошибки не роняют всю обработку

**Компромисс:** Два токена вместо одного, но haiku дёшевый.

## [2026-03] gpt-4o-mini-transcribe вместо Gladia

**Решение:** Вернулись на OpenAI Whisper (gpt-4o-mini-transcribe) после эксперимента с Gladia.

**Почему:**
- Gladia требовал polling-цикла (до 3 мин ожидания), сложная логика retry
- gpt-4o-mini-transcribe работает проще: загрузил файл → получил текст
- Gladia имел преимущество в диаризации (разделение спикеров), но качество румынского/русского оказалось сопоставимым
- Меньше внешних зависимостей

**Что потеряли:** Диаризацию (определение спикеров). Пока не критично — Claude и так понимает контекст.

## [2026-03] is_first_contact: разные шаблоны резюме

**Решение:** При первом звонке — полный шаблон с профилем клиента. При повторных — только дельта.

**Почему:**
- Менеджер уже знает профиль клиента. Дублировать имя/бюджет/запрос в каждой заметке — засоряет CRM.
- Дельта-шаблон значительно короче, читается быстрее.
- `isFirstContact = client.total_calls === 0` — простая и надёжная проверка.

**Риск:** Если первый звонок обработан с ошибкой, второй звонок будет считаться повторным и выдаст дельту без полного профиля. Пока принимаемо.

## [2026-03] Тег "Квал" вместо температурных тегов

**Решение:** Только один тег `"Квал"` в AmoCRM, только при первом контакте.

**Почему:**
- Температура меняется от звонка к звонку — держать её в тегах неинформативно и создаёт мусор
- "Квал" — бинарный, стабильный, полезный для фильтрации в воронке
- Неквал-лиды тег не получают — это тоже сигнал (нет тега = неквал или ещё не обработан)

**Альтернатива:** Температура в кастомном поле сделки. Можно реализовать позже.

## [2026-03] RAG-контекст в анализе звонков

**Решение:** Перед анализом делаем semantic search по первым 500 символам транскрипта, топ-3 чанка передаём в оба промпта Claude.

**Почему:**
- Менеджеры иногда называют неверные цены или условия — Claude с контекстом из базы знаний это замечает
- База знаний (прайс-листы, условия рассрочки) обновляется без изменения кода
- 500 символов = достаточно для semantic match по теме звонка

**Компромисс:** +200-500мс на каждый анализ. Приемлемо.

## [2026-03] Supabase вместо отдельного PostgreSQL

**Решение:** Supabase как единый бэкенд для БД, хранилища файлов и векторного поиска.

**Почему:**
- pgvector встроен — не нужен отдельный векторный стор (Pinecone, Weaviate)
- Storage bucket — не нужен S3
- Один сервис, один ключ, один dashboard
- Service role ключ позволяет обходить RLS — упрощает бэкенд (RLS планируется позже)

## [2026-03] Express вместо Next.js API routes для бэкенда

**Решение:** Отдельный Express-сервер на Railway, а не API routes внутри Next.js.

**Почему:**
- Webhook от PBX должен быть доступен 24/7, независимо от фронтенда
- Обработка звонка (скачать MP3, транскрибировать, два AI-вызова) занимает 30-90 сек — Vercel функции таймаутятся
- Railway даёт долгоживущий процесс и прямой контроль над памятью/файлами (`/tmp`)

## [2026-03] Admin-панель только для RAG (не для звонков)

**Решение:** Фронтенд (`leapy-admin`) управляет только базой знаний RAG. Просмотр звонков и оценок — пока только через Supabase dashboard.

**Почему:**
- MVP-приоритет: RAG нужен срочно, дашборд со звонками — позже
- Простота: не нужна авторизация, нет чувствительных данных

**Запланировано:** Дашборд с оценками менеджеров, профили клиентов.

## [2026-03] Монорепо + отдельный репо фронтенда

**Решение:** Код `admin/` живёт и в основном репо `Leapy`, и в отдельном `leapy-admin`.

**Почему:** Vercel требует отдельный репо для деплоя Next.js без монорепо-конфига. Временное решение.

**Проблема:** Нужно синхронизировать изменения вручную между двумя репо. Технический долг.

**Будущее решение:** Настроить Vercel root directory на `admin/` в основном репо и избавиться от дублирования.
